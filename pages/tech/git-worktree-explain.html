AI駆動開発における Git Worktree 運用まとめ

GitHub Copilot 等の AI を用いて並列実装を行う際の、 Git Worktree の役割・通常の branch との違い・実運用パターンを整理したメモ。

1. AI並列開発で何が問題になるのか

通常の Git 運用と AI 駆動開発では、問題の種類が少し異なる。

主な問題

問題 内容
Git conflict 同じファイルを複数 branch が変更する
作業空間汚染 AI が他タスクの変更を見て誤実装する
Context 汚染 Copilot が無関係 diff や WIP を拾う
Branch 切替事故 checkout により他作業が壊れる
AI 開発で最悪の構成 
1 working directory
+
複数AI
+
複数タスク

2. 通常の branch 運用の限界

通常は以下のような構成。

project/
 ├ .git
 ├ src
 └ ...

この場合、working directory は 1 つしか存在しない。

git checkout feature/auth
git checkout feature/device

branch を切り替えるたびに、作業空間全体が切り替わる。

VSCode を複数開いていても、同じ directory を見ているため、 branch 切替が全 Window に影響する。

3. Worktree とは何か

Git Worktree は、

1つの repository に対して、複数の working directory を作る仕組み

である。

通常

project/

Worktree

project-main/
project-auth/
project-device/

ただし repository は共有。

shared .git repository
  ├ project-main
  ├ project-auth
  └ project-device

4. branch と worktree の違い

項目 通常 branch worktree
working directory 1つ 複数
branch切替 checkoutで切替 同時存在可能
VSCode状態 共有 分離
AI context 混線しやすい 分離しやすい

5. clone と worktree の違い

項目 clone worktree
.git 別 共有
fetch 個別 共有
disk usage 重い 軽い
Git管理 別repo 同一repoとして管理

重要な点

worktree の本質は「軽量 clone」ではなく、

Git が「複数作業空間」を正式サポートしている

という点。

6. 実際の worktree 作成方法

worktree 作成

cd project

git worktree add ../project-auth -b feature/auth

結果

project/
project-auth/

VSCode

code ../project-auth

7. ディレクトリ構成パターン

パターンA: 横並び（一般的）

workspace/
 ├ repo-main/
 ├ repo-auth/
 ├ repo-device/
 └ repo-ui/

メリット

  • VSCode と相性が良い
  • AI context が混線しにくい
  • 検索や watcher が安定

パターンB: repo配下

workspace/
 └ repo/
     ├ .git
     └ trees/
         ├ auth/
         ├ device/
         └ ui/

問題点

  • VSCode 検索が混線
  • Copilot context 汚染
  • Docker build context に混ざる
  • eslint/prettier が二重走査
人間視点では整理されて見えるが、 AI/IDE/tooling 視点では分離された方が安全。

8. おすすめ構成

workspace/
 ├ main/
 │   └ repo/
 │
 └ worktrees/
     ├ repo-auth/
     ├ repo-device/
     └ repo-ui/

理由

  • main と task workspace を分離できる
  • AI context 汚染を減らせる
  • 管理しやすい

9. AI駆動開発での運用ベストプラクティス

AI 1チャット = 1責務

悪い例 
ログイン修正しながら
テーブルUIも修正
良い例 
chat-auth = auth専用
chat-ui = ui専用

AI に責務を毎回明示

このチャットは device module 専用。
他moduleは変更しない。

Diff review 前提

生成
↓
diff review
↓
commit

AI生成コードは必ず review する。

10. コンフリクトはなくならない

worktree は conflict 解決ツールではない。

worktree が解決するのは 「状態分離」「context分離」「作業空間分離」。

同じファイルを変更すれば conflict は起きる。

11. 実際の現場で多い運用

  • 小さいPRを大量に出す
  • branch寿命を短くする
  • feature toggle を使う
  • contract first(API schema固定)
  • interface先行

12. 本質

本当に重要なのは worktree そのものではない。

重要なのは以下。

  • 状態分離
  • 責務分離
  • context分離
  • AIごとの workspace 分離

worktree はそれを実現するための、 Git公式のやり方である。
